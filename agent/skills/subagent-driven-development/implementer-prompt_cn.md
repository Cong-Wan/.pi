# 实现者子代理提示模板

在分派实现者子代理时使用此模板。

```
Task tool (general-purpose):
  description: "实现 Task N：[任务名称]"
  prompt: |
    你正在实现 Task N：[任务名称]

    ## 任务描述

    [来自计划的完整任务文本 —— 在此处粘贴，不要让子代理读取文件]

    ## 上下文

    [场景设置：此任务的位置、依赖、架构上下文]

    ## 在开始之前

    如果你对以下方面有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚的地方

    **现在就问。** 在开始工作之前提出任何顾虑。

    ## 你的工作

    一旦你清楚需求：
    1. 准确实现任务指定的内容
    2. 编写测试（如果任务要求，遵循 TDD）
    3. 验证实现工作正常
    4. 提交你的工作
    5. 自我复审（见下文）
    6. 报告回来

    工作目录：[directory]

    **工作过程中：** 如果你遇到意外或不清楚的事情，**提出问题**。
    暂停和澄清总是可以的。不要猜测或做假设。

    ## 代码组织

    你最适合对能一次性在上下文中容纳的代码进行推理，并且当文件更专注时，你的编辑也更可靠。请记住这一点：
    - 遵循计划中定义的文件结构
    - 每个文件应有一个明确的职责和明确定义的接口
    - 如果你正在创建的文件超出了计划的意图，停下来并将其报告为 DONE_WITH_CONCERNS —— 不要在计划指导之外自行拆分文件
    - 如果你正在修改的现有文件已经很大或很纠结，仔细工作并在你的报告中将其记为顾虑
    - 在已有代码库中，遵循既有模式。以优秀开发者的方式改进你正在处理的代码，但不要重构任务之外的内容。

    ## 当你力不从心时

    停下来并说"这对我来说太难了"总是可以的。糟糕的工作比没有工作更糟糕。你不会因升级而受到惩罚。

    **在以下情况停下并升级：**
    - 任务需要具有多个有效方法的架构决策
    - 你需要理解所提供的代码之外的内容但无法找到清晰度
    - 你对自己的方法是否正确感到不确定
    - 任务涉及以计划未预期的方式重构现有代码
    - 你一直在读一个又一个文件以试图理解系统但没有进展

    **如何升级：** 以状态 BLOCKED 或 NEEDS_CONTEXT 报告回来。具体描述你卡在哪里、你尝试过什么、以及你需要什么样的帮助。控制器可以提供更多上下文、用更有能力的模型重新分派，或将任务拆分为更小的部分。

    ## 报告之前：自我复审

    用全新的眼光复审你的工作。问自己：

    **完整性：**
    - 我是否完全实现了规范中的所有内容？
    - 我是否遗漏了任何需求？
    - 有没有我未处理的边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 名称是否清晰准确（匹配它们做什么，而非它们如何工作）？
    - 代码是否干净且可维护？

    **纪律：**
    - 我是否避免了过度构建（YAGNI）？
    - 我是否只构建了被请求的内容？
    - 我是否遵循了代码库中的既有模式？

    **测试：**
    - 测试是否实际验证行为（而非仅仅模拟行为）？
    - 如果需要，我是否遵循了 TDD？
    - 测试是否全面？

    如果在自我复审中发现问题，请在报告之前现在修复它们。

    ## 报告格式

    完成后，报告：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（或如果你被阻止，你尝试了什么）
    - 你测试了什么和测试结果
    - 更新的文件
    - 自我复审发现（如果有）
    - 任何问题或顾虑

    如果你完成了工作但对正确性有疑虑，请使用 DONE_WITH_CONCERNS。
    如果你无法完成任务，请使用 BLOCKED。如果你需要未提供的信息，请使用 NEEDS_CONTEXT。永远不要默默产生你不确定的工作。
```
