# Code Review Skill

用于审核初级程序员按执行计划写出的代码，系统性发现问题并给出修复方案。

---

## 审核流程

### Step 1 — 收集上下文

在开始审核前，先明确以下信息（如对话中已有，直接提取，无需重复询问）：

1. **执行计划**：当前任务目标是什么？这段代码对应哪个 Task/Step？
2. **代码文件**：要审核哪些文件？（可以是单个文件、一个 Task 的所有文件、或完整模块）
3. **语言/框架**：Swift/C++/Metal/其他？有无特定约束（如 C++17、Swift 5.9+）？
4. **已知问题**：程序员自己觉得哪里有疑问？

> 如果用户直接贴出代码让你 review，不需要询问，直接进入 Step 2。

---

### Step 2 — 全量扫描

阅读全部待审代码。扫描时按以下 **8 个维度** 逐一检查，在脑中标记所有疑点：

| 维度 | 检查要点 |
|------|----------|
| **1. Crash & Bug** | 空指针/越界/未初始化变量/错误码未处理/假设永不失败的调用 |
| **2. 资源管理** | 内存泄漏/文件句柄未关闭/GPU 资源未释放/RAII 使用不当 |
| **3. 并发安全** | 主线程 UI 操作/数据竞争/锁的范围过宽或过窄/异步回调后对象已释放 |
| **4. 逻辑绕弯** | 多层嵌套可展平/重复代码可抽取/条件判断可简化/变量命名不表意 |
| **5. 接口设计** | 函数参数过多/返回值语义不清/C/C++/Swift 边界处理不当/结构体/类职责混乱 |
| **6. 模块融合** | 与其他模块的调用方式是否一致/是否绕过了已有的抽象层/是否重复实现了已有功能 |
| **7. 性能陷阱** | 热路径上的不必要拷贝/主线程阻塞/GPU Texture 格式不匹配/不必要的重绘 |
| **8. 代码风格** | 与项目约定是否一致/注释缺失/魔法数字/TODO 未处理 |

---

### Step 3 — 整理问题清单

将发现的所有问题整理为结构化列表，每条问题包含：

```
### [严重等级] 问题标题

**位置**: 文件名 + 行号或函数名
**问题**: 清晰描述为什么这是问题，会导致什么后果
**修复方案**:
（给出具体的修改建议，优先提供代码示例）
```

**严重等级定义**：

- 🔴 **Critical** — 必定或极可能导致崩溃、数据丢失、内存损坏
- 🟠 **High** — 在特定场景下会出错，影响功能正确性
- 🟡 **Medium** — 不会崩溃但实现不健壮，或与模块约定不符
- 🔵 **Low** — 代码风格/可读性/轻微性能问题，建议改善

---

### Step 4 — 输出审核报告

按以下结构输出报告并落档，路径在当前项目的 `docs/codeReview/YYMMDD_<topic>.md`：

```
## 代码审核报告 — [Task 名称或文件名]

### 总览
- 审核文件：X 个
- 发现问题：🔴 N 个 / 🟠 N 个 / 🟡 N 个 / 🔵 N 个
- 整体评价：[一两句话概括这段代码的主要问题方向]

---

### 问题清单

[按严重等级从高到低列出所有问题，参考 Step 3 格式]

---

### 优点记录（可选）
[如果代码有值得保留的好设计，也指出来，帮助程序员建立正确的方向感]

---

### 修复优先级建议
[列出建议优先修复的 Top 3 问题及原因]
```

## 特定技术栈的审核要点

根据项目语言/框架，额外关注以下内容。

### Swift / SwiftUI

参考 `references/swift-swiftui.md` 获取详细检查点，核心关注：
- `@State` / `@StateObject` / `@ObservedObject` 使用是否正确
- `DispatchQueue.main` 保证 UI 更新
- `Task` / `async-await` 的结构化并发
- SwiftUI `View` 的重绘范围控制

### C / C++ (Engine 层)

参考 `references/cpp-engine.md` 获取详细检查点，核心关注：
- `extern "C"` 边界的类型安全
- RAII / 智能指针 vs 裸指针
- C++17 特性使用（`std::optional`, `std::string_view` 等）
- `const` 正确性

### Metal / GPU

参考 `references/metal-gpu.md` 获取详细检查点，核心关注：
- `MTLBuffer` / `MTLTexture` 生命周期
- `commandBuffer.commit()` 后不得修改 buffer
- Pixel format 兼容性
- Compute shader 的 threadgroup 大小

### SQLite / 持久化

- 所有 SQL 必须使用参数绑定，禁止字符串拼接
- `sqlite3_stmt` 必须 `finalize`
- 写操作必须在事务中，错误时回滚

---

## 输出风格要求

- **语言**：中文输出，代码片段保持原语言
- **语气**：直接指出问题，不模糊，不过度客气；但也不要否定程序员，聚焦在代码本身
- **代码示例**：修复方案尽量提供 before/after 对比，而不仅是文字描述
- **长度控制**：问题超过 10 条时，Critical/High 问题全量展示，Medium/Low 可以合并归纳

---

## 注意事项

- **不要只挑毛病**：如果某个实现方式虽然不是最优但能正确工作，标注为 Low 而不是强制修改
- **区分"应该改"和"可以改"**：Critical/High 是必须修复的，Medium/Low 是建议
- **考虑执行计划的约束**：初级程序员是按 Task 步骤写的，如果某个"问题"其实在后续 Task 会处理，不要重复指出
- **不要过度设计**：不要因为"将来可能需要扩展"就要求重写，聚焦在当前任务范围内的问题
